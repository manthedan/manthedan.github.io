Chengyu Search — https://findchengyu.com/ · GitHub

A search engine for Chinese idioms.

Last year I spent a month in Hangzhou studying Chinese, and I realized a lot of students were missing out on the cultural lessons of the language that I had learned through Ricky. Dictionaries have the idioms, but you would only find them if you are directly looking for them.

I figured there should be a semantic ML approach, so I did a large amount of data processing on some open source Chinese dictionaries, and created an embedding search.

There's still much room for optimization if this ever needed to scale, but as a focused learning tool it runs impressively well with simple cosine similarity matching on a conventional CPU server.

Highlights

  • Hybrid semantic + lexical search over 5,900+ CC-CEDICT-backed chengyu
  • Auto-routed English, Hanzi, and pinyin queries
  • Fuse.js lexical retrieval plus Transformers.js / MiniLM semantic ranking
  • Local embedding artifacts and cache-aware Express API
  • Learner UX: simplified/traditional toggle, saved idioms, Anki TSV export, and clickable example-word tooltips

0x88 — https://0x88.app/ · GitHub

A browser-native chess engine platform. It runs engines locally in the browser with WebAssembly and WebGPU experiments, with play, analysis, and engine-vs-engine arena modes. This project became a deep systems exercise in browser compute, engine packaging, model artifacts, and performance testing.

The most interesting neural chess engines require some technical background to use. My goal was to see if I could remove that completely by getting them to run in the browser, similarly to https://webllm.mlc.ai/ for LLMs.

To that degree I have been fairly successful. They are quite usable on an Apple Silicon Mac, and will become increasingly usable as more WebGPU/WebNN capabilities are promoted by the Chromium / browser vendor teams.

I experimented with several runtime approaches, including hand-rolling WGSL shaders. At present, the ONNX runtimes are generally the most stable and usable.

Most chess engines are still CPU optimized, so I have created standard procedures for porting them in browser using WASM and Relaxed SIMD targets to approximate as much of local hardware performance as browsers allow.

Highlights

  • Browser-first chess platform with Play, multi-engine Analysis, and Arena modes
  • Local WebGPU/WASM execution with cached engine and model artifacts
  • LC0, Maia, Stockfish, Berserk, PlentyChess, Viridithas, and Reckless runtime work
  • ONNX Runtime WebGPU inference with QDQ, WGSL, TVMJS, and WebNN research lanes
  • Multi-engine analysis, PGN review, local IndexedDB game storage, and arena scheduling
  • Artifact provenance, parity testing, and GPL/AGPL corresponding-source packaging

SweetSpot — https://github.com/manthedan/sweet-spot

The idea here is quite simple: there are many times where one needs cheap distributed compute. AWS Spot Instances are cheap because they run on spare EC2 capacity, but they can be interrupted when AWS needs that capacity back.

SweetSpot treats workers as volatile and makes task completion explicit. A bounded-parallelism approach uses AWS Batch Spot instances as volatile workers, while SQS coordinates at-least-once task execution and S3 stores outputs with deterministic done markers.

The design goal is to augment the areas where agents struggle. In my experience, if you let Claude or GPT build out your ETL, they are too conservative where you want them to press (throughput) but have no sense of your own real-world sensitivities (cost, wall time, job batch size). So I've tried to design this in a way that is usable by agents, tries to leverage more cost-efficient ARM instances, and is extremely durable and empirical, placing testing and experimental verification ahead of agent instincts.

Highlights

  • Python CLI/framework for idempotent AWS Batch Spot workloads
  • SQS task queue with S3 outputs, summaries, and deterministic done markers
  • At-least-once execution with Spot retry handling, DLQ redrive, and duplicate-attempt validation
  • Canary-first sizing and telemetry-driven x86 / ARM / Graviton cost modeling
  • OpenTofu infrastructure and dry-run-first operational commands

Centipawn — open weights, playable on 0x88.app, not yet open source

A browser-first neural chess engine in the spirit of Leela Chess Zero, but drastically compressed: where LC0's transformer models run 140M+ parameters, Centipawn operates at under 1M. Early results suggest grandmaster-level or stronger play, though I am evaluating strength through protocol-specific engine benchmarks rather than a single Elo claim. In the world of top chess engines this is still modest, but the goal is not to compete on raw strength -- it is to build a model small enough to inspect, deploy, and iterate on entirely in the browser. Architecture diagrams and evaluation write-ups are in progress on the 0x88 docs page.

Highlights

  • Browser-first Leela-style model/search stack
  • Compact CNN and Move / Board-Square token transformer architectures
  • Policy and WDL/value heads, with action-value and regret experiments
  • TypeScript/ONNX evaluator path integrated with PUCT search
  • Classic PUCT, AV-PUCT, and search-mode arena tooling
  • Frontier-card evaluation across strength, size, latency, blunders, and calibration
  • LC0 search-distillation and self-play infrastructure for stronger future checkpoints
